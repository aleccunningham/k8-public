Drone

Drone CI Server and Agent configuration

Location: https://drone.kubernetes.lol/

Official documentation can be found at docs.drone.io. This includes Plugin, CLI, and API references

The following variables must be set for this to run correctly:

CLI use

  • DRONE_SERVER=https://drone.kubernetes.lol/
  • DRONE_TOKEN=<found in drone server settings>

Hosted

  • DRONE_GITHUB=true
  • DRONE_GITHUB_URL=https://github.com
  • DRONE_GITHUB_CLIENT=<github oatuh>
  • DRONE_GITHUB_SECRET=<github oatuh>
  • DRONE_GITHUB_SCOPE=repo,repo:status,user:email,read:org
  • DRONE_SECRET=<string>

Using the CLI

One of Drones features the differentiate it from other CI services is its CLI client. After configuring the local agent via setting the DRONE_SERVER and DRONE_TOKEN variables, enter a directory in which a .drone.yml file exists you can experiment; in particular, being able to run drone exec to run the pipeline locally (including supporting services and every other configuration option available to drone pipelines).

Secrets

Drone encrypts and stores secrets defined through the CLI client via STDIN or file, and handles them extremely well. To see what secrets are tied to a repository, run:

$ drone secret add \
  --repository aleccunningham/chatops-ai \
  --name aws_client_token \
  --value <string>

The secret can then be inserted at buildtime via the capitalized version of the secrets string (so in this case, $AWS_CLIENT_TOKEN).

You can also limit where the variable can be used via the --event flag (Note You must set --event pull_request to allow injecting of variables during PR builds), along with --image, tying a secret to allow its use on an image by image basis

Deploy & Jenkins

By setting conditional statements in the pipelines such as

when:
  environment: prodref
  branch: master

Drone will wait until a build has been “promoted” via drone deploy (in this case, promoted to the prodref environment) to run any commands defined in the jobs. Drone deploy is run via CLI and the format is as follows:

$ drone deploy <repo> <build_number> <environment>

This manual step is great for handling independent environments. The build number can be found on the repositories Drone page – just deploy the particular build that you would like to promote.

You can find full documentation for the CLI tool here. If you’re not already taking advantage of it (and you’re working on a project that currently supports drone), you’re missing out!

Our drone yamls use a jenkins plugin to trigger two jobs - deploy-prodref and deploy-prod depending on the environment the job is promoted to. Below is an example of this plugin, along with pushing images to GCR:

gcr:
  image: plugins/gcr
  registry: us.gcr.io/$GCLOUD_PROJECT
  repo: aleccunningham/django-react
  tags: ["latest", "$BRANCH"]
  secrets: [ google_credentials ]

jenkins_prodref:
  image: appleboy/drone-jenkins
  url: http://example.com
  user: appleboy
  token: xxxxxxxxxx
  job: drone-jenkins-plugin-job
  when:
    status: [ success ]
    event: deployment
    environment: prodref

jenkins_prod:
  image: appleboy/drone-jenkins
  url: http://example.com
  user: appleboy
  token: xxxxxxxxxx
  job: drone-jenkins-plugin-job
  when:
    status: [ success ]
    event: deployment
    environment: prod
